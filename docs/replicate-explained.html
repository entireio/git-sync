git-sync replicate — How It Works

Source-authoritative, relay-only mirroring · annotated walkthrough with file references

The contract in one line: make the target's refs match the source exactly, by streaming packs directly between the two remotes — and if that can't be done purely via relay, fail loudly rather than fall back to anything local. No fast-forward gate, no force flags, no materialized fallback.

3Planning — what action each ref gets

PlanReplicationRef planner.go:365 compares each desired source ref to the target's current value. The defining trait: no ancestry / fast-forward check — a divergent ref simply becomes an unconditional overwrite.

ConditionActionReason string
source hash == target hashSKIP<hash> already current
target ref absent / zeroCREATE<hash> -> <new>
differs (any direction)UPDATE<old> -> <new> (replicate overwrite)
on target, absent on source (--prune only)DELETE<hash> -> <deleted>
Why no force flags? Overwrite is the baseline, so there's no fast-forward gate for a --force-with-lease/--force-blind flag to open. The syncer rejects them outright (syncer.go:646): "replicate does not support force flags; use sync instead."

4Execution — the two-phase relay

Execute splits plans into updates (create + update) and deletes, then runs two phases. git-sync sits in the middle and never decodes the object graph — the pack streams straight through.

Why deletes go last & separate

A delete needs no objects, so sending them after the pack push keeps the object-bearing relay clean. PushPack actively rejects delete commands (push.go:526).

6Empty-target detour → bootstrap

When every managed target ref is absent, replicate hands off to the bootstrap strategy bootstrapWithInputs · syncer.go:1121 with reason empty-target-managed-refs.

HEAD-first ordering

hoistSourceHeadPlan pushes the source's HEAD branch first, so hosts that infer the default branch from the first push (GitHub, GitLab) end up matching the source automatically. bootstrap.go:202

Batched bootstrap

With --target-max-pack-bytes, the chain is walked and split into checkpoint sub-packs that each fit the limit — first-parent (default) or topo ordering. bootstrap.go:255
