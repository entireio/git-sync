How git‑sync works

A remote-to-remote Git mirroring tool that streams packs between two servers — without ever creating a local clone. From the one core idea down to every knob, failure mode, and the SRE playbook.

01 — Core idea

Move Git data between two remotes, without a local copy

The normal way to mirror one Git remote into another is: clone --mirror to disk, then push --mirror back up. That works, but it turns a remote-to-remote operation into a local storage and local bandwidth problem — every byte lands on your disk twice.

git-sync's one big bet is relay: fetch a packfile from the source and stream those exact bytes straight into the target's receive-pack, without decoding the object graph locally. The process holds at most one pack chunk at a time — never the whole repo.

The whole product in one line git-sync is the missing middle layer: provider-agnostic, remote-to-remote, automation-friendly, and explicit about what is safe to relay vs. what needs a fallback.

When it's the right tool

  • Strong fit: migrations and repeat sync jobs where updates are mostly fast-forward, on disposable runners with no persistent repo storage.
  • Weak fit: workflows that need arbitrary local Git reconciliation, history rewriting, or already keep warm local mirrors. A local-clone service is the more general model; git-sync is the lower-state, operationally cheaper one.
02 — Basics

Four things you need to picture

Everything downstream is built from these. If you already know Git internals, skim — but the have/want handshake is the load-bearing one.

1. Refs are just named pointers

A ref (refs/heads/main, refs/tags/v1) is a name pointing at a 40/64-char hash. Mirroring a repo is really just: make the target's ref pointers match the source's, and ship whatever objects the target is missing to back them.

2. Objects travel in a packfile

Commits, trees, blobs and tags are content-addressed objects. Git ships them in bulk as a packfile — a compressed, delta-encoded stream that starts with a 12-byte header:

"PACK"  uint32 version  uint32 object_count

git-sync peeks at exactly those 12 bytes to estimate a pack's size before committing to a multi-gigabyte transfer (more in §3).

3. have / want is the negotiation

To fetch incrementally, the client tells the source what it already has and what it wants:

want 9f2c…  # a source tip I need
have 41ab…  # a hash the TARGET already has
done
The clever bit git-sync advertises the target's current tips as have lines to the source. The source then computes a pack containing only the objects the target lacks — so reruns download less the more history the two sides already share.

4. Two services per remote

Smart HTTP exposes two RPC services. Discovery (a GET) lists refs & capabilities; the RPC (a POST) carries the actual fetch or push body.

service
role
git-upload-pack
SOURCE — list refs, serve objects (fetch)
git-receive-pack
TARGET — list refs, accept objects (push)

git-sync is smart-HTTP-only (plus SSH) — dumb HTTP can't do the capability discovery and have/want exchange that relay depends on.

03 — The decision

How a single sync picks its path

This is the heart of the engine. Every run plans every ref first — classifying each into create / update / delete / skip / block — then chooses one of four transfer modes. Three of them are relay (cheap, streaming); the fourth materializes objects in memory and is deliberately bounded.

Bootstrap relay · empty target

Target has none of these refs yet. Stream the source pack straight in. sync does this automatically — same command seeds and maintains. The source HEAD's branch is pushed first so the mirror's default branch matches.

memory: streaming · one chunk at a time
Incremental relay relay · fast-forward

Every ref is a clean fast-forward branch update, a branch create, or a brand-new tag. The narrow fast path — relays the source pack with target tips as have. Never decodes objects locally.

memory: streaming
Replicate relay · overwrite

Source-authoritative: target refs are made to match source unconditionally (create/update, even divergent). Relay-only — if it can't relay, it fails and tells you to use sync rather than silently materializing.

memory: streaming · tolerates no-thin targets
Materialized fallback in-memory · bounded

Anything relay can't safely do — force pushes, prune, deletes, tag retargets, or a no-thin target under sync. Decodes the needed objects into an in-memory go-git store, computes closure, and re-encodes a pack. Guarded by --materialized-max-objects.

memory: scales with objects to send — the risky path
Memory model — the real predictor "Repo size" alone tells you nothing. For the fallback, what matters is: how many objects must be sent, how big the missing blobs are, and how much the target already overlaps. A small branch delta on top of an existing tip is cheap; a broad fallback with no shared history can approach the whole pushed object set in RAM.

The lazy-fetch optimization

git-sync won't download the source closure unless planning actually needs it. Pure skip/create plans that take incremental relay skip the upfront fetch entirely — relay does its own fetch on the source. The closure is pulled only for ancestry checks (FF detection on a divergent branch) or when the materialized fallback ends up running:

// internal/syncer/syncer.go
func needsLocalSourceClosure(cfg, desired, targetRefs) bool {
    if cfg.ForceAny() || cfg.Prune {
        return true   // fallback will run; it needs the closure
    }
    for targetRef, want := range desired {
        targetHash := targetRefs[targetRef]
        if targetHash.IsZero() { continue }  // create
        if targetHash == want.SourceHash { continue } // skip
        return true   // divergent → need ancestry data
    }
    return false  // all skip/create → relay handles it, no fetch
}
04 — On the wire

What relay actually looks like

Going deep now. Every smart-HTTP body is a sequence of pkt-lines: a 4-byte hex length prefix then payload. Three lengths carry no payload and act as control markers.

markerhexmeaning
flush0000end of section / stream
delim0001section delimiter (v2)
end0002end of v2 response

Sideband: three channels in one stream

When the server offers side-band-64k, the response is multiplexed. The first byte of each payload selects the channel:

  • 0x01 — pack data (the bytes we relay)
  • 0x02 — progress text → stderr (or discarded)
  • 0x03 — fatal error → fail the run

The relay frame swap

This is the conceptual move. The source's pack chunks arrive one-per-pkt-line on channel 0x01. git-sync strips the pkt-line + sideband envelope and concatenates the raw pack bytes into the target's push body, right after the command list and a flush. Byte-for-byte, the source-produced pack shows up on the target's wire — never re-encoded.

Why no-thin matters

A thin pack contains deltas whose base objects live outside the pack — smaller on the wire, but the receiver must resolve them against its existing DB. git-sync never requests thin-pack on the source, so every relayed pack is self-contained. That's why relay is safe against any target. The paths still differ on policy:

  • Replicate proceeds on no-thin targets — the pack is self-contained, so it's safe.
  • Incremental relay (inside sync) is conservative: it skips no-thin and falls back to materialized. A safety margin, not a hard requirement.
  • Bootstrap ignores no-thin entirely — empty target, always fine.

Pre-flight pack sizing (batched bootstrap)

For huge initial migrations, git-sync reads just the 12-byte PACK header, multiplies object_count by a per-object estimate, and if the projected pack exceeds --target-max-pack-bytes, it aborts the fetch and inserts a checkpoint — wasting 12 bytes instead of gigabytes. Body-too-large rejections after the fact insert a midpoint and retry. Both converge in O(log n) splits.

Protocol v2, where it helps

Source-side discovery and fetch use v2 when available; push always stays on v1 receive-pack. --protocol auto tries v2 then falls back; v2 requires it; v1 skips the attempt. v2 buys cheaper ref listing (ls-refs with prefixes) and filter=tree:0 commit-graph-only fetches that batched-bootstrap planning relies on.

05 — Commands

The surface you actually type

sync

Mirror source → target. Auto-bootstraps empty targets, fast-forward by default. The everyday command.

replicate

Source-authoritative overwrite via relay. Fails rather than materialize locally.

plan

Dry run. Same flags as sync; --mode replicate previews a replicate.

probe / fetch / bootstrap

Lower-level inspection & seeding. Available via --help / the unstable library.

The quick start

git-sync sync \
  --source-token "$GITSYNC_SOURCE_TOKEN" \
  --target-token "$GITSYNC_TARGET_TOKEN" \
  https://github.com/src-org/repo.git \
  https://github.com/dst-org/repo.git

Common shapes

# specific branches, with a rename
git-sync sync --branch main,release --map main:stable SRC DST

# tags + prune refs that vanished from source
git-sync sync --tags --prune SRC DST

# everything, minus noisy PR refs
git-sync sync --all-refs --exclude-ref-prefix refs/pull/ SRC DST

# overwrite, fail loudly if relay impossible
git-sync replicate --stats SRC DST
Force, the modern way The old --force is gone. Use --force-with-lease (allow non-FF, but reject if the target moved during the run — a per-run lease, no state between runs) or --force-blind (overwrite unconditionally, like git push --force). Mutually exclusive. replicate and bootstrap take neither.

Auth resolution order

  • explicit flags (--source-token, --source-bearer-token…)
  • GITSYNC_* env vars
  • git credential fill helper (http/https only; deferred until a 401)
  • anonymous

For GitHub-style hosts, basic auth with the token as the password (username=git). Bearer tokens supported via *_BEARER_TOKEN.

06 — git-sync for SRE

Operating it in anger

git-sync is built for disposable runners and automation. The operational story rests on three pillars: plan before you push, read the typed output, and tune the guardrails per target.

1 — Plan first, always

plan takes the same flags as the real run and shows the exact per-ref actions plus which relay mode would fire — without touching the target. Wire it into CI as a gate. If sync finds blocked refs (non-FF without a force flag), it exits non-zero before pushing anything, so a partial mirror is never the failure mode.

2 — Consume the JSON, not the text

Add --json to anything. The interface is stable (camelCase, hashes as strings). Key fields to alert on:

{
  "pushed": 12, "skipped": 40, "blocked": 0,
  "deleted": 1,  "warned": 0,
  "relay": true, "relayMode": "incremental",
  "relayReason": "fast-forward-branch-or-tag-create",
  "batching": false, "batchCount": 0,
  "protocol": "v2",
  "execution": { "sourceHead": "refs/heads/main" }
}
  • blocked > 0 → run failed; a ref needs --force-with-lease or human eyes.
  • relay:false on a job you expected to be cheap → you fell into the materialized fallback. Investigate why via relayReason (e.g. incremental-disabled-by-force-prune-or-dry-run, no-thin).
  • warned > 0 → best-effort run had per-ref rejections (e.g. GitHub refusing refs/pull/*). Expected with --all-refs; surprising otherwise.

3 — Measure memory & throughput

--measure-memory samples elapsed time and Go heap; --stats adds per-service request/byte/want/have/command counters and a throughput line. Use these to set --materialized-max-objects from real data instead of guessing.

measurement: elapsed-ms=8421 peak-heap-inuse-bytes=214958080 gc-count=11
stats: git-upload-pack fetch requests=1 response-bytes=98231140 wants=7
Bounded by design info/refs responses are capped at 64 MiB and buffered RPC responses at 128 MiB — the streaming relay body bypasses these caps because it reads pack chunks incrementally. The local process is protected even against a hostile or broken remote.

4 — Huge first migrations: batch it

When the target enforces a request-body limit, set --target-max-pack-bytes. git-sync plans the bootstrap as a chain of checkpoints, each sub-pack sized to fit, and auto-subdivides on the fly. Two orderings:

first-parent (default)

Walks only the trunk backbone. Cheapest enumeration; fine for most repos.

topo

Every reachable commit is a candidate checkpoint. For merge-heavy repos where one first-parent step drags in a huge side branch that can't be subdivided.

topo's hidden requirement Under topo, the internal refs/gitsync/bootstrap/* temp ref may take non-fast-forward updates between checkpoints. Targets that enforce receive.denyNonFastforwards across all refs (not just refs/heads) will reject them. Major hosts (GitHub, GitLab, Bitbucket, Cloudflare) don't, by default. User-visible refs still get a single FF update at cutover.

5 — Default branch & ordering

Bootstrap pushes the source HEAD's branch first, so hosts that infer the default from the first push (GitHub, GitLab) get the right default branch automatically. Git's wire protocol can't set a remote symref, so for raw bare repos you set HEAD out of band: git init --bare --initial-branch=… beforehand, or the host API afterward (PATCH /repos/{o}/{r} with default_branch).

6 — Provider quirks worth pre-configuring

  • Ref-update batch size: --target-max-ref-updates caps commands per push (default 5000). Raise for entire-server targets (≤25000); lower for providers that reject large ref pushes.
  • Regional redirects: some hosts 307 /info/refs to a replica. Set --source/--target-follow-info-refs-redirect so RPC POSTs follow to the redirected host (avoids sticky-session 404s).
  • SSH: shells out to your ssh binary with BatchMode=yes — uses ~/.ssh/config, no prompts. Pre-seed known_hosts or set StrictHostKeyChecking=accept-new. Throughput stats are not yet byte-counted over SSH.

A safe production loop

# 1. gate on a dry run
git-sync plan --json SRC DST | jq '.blocked'   # must be 0

# 2. real run, bounded + observable, fail closed
git-sync sync --json --stats --measure-memory \
  --materialized-max-objects 500000 \
  --target-max-pack-bytes 536870912 \
  SRC DST > result.json || alert "sync failed"

# 3. assert the path you expected
jq 'select(.relay==false) | "fell back to materialized!"' result.json
07 — The knobs

Every flag that changes behavior

Scope — what gets mirrored

--branch a,b,c
comma list; default is all source branches
--map src:dst
rename a ref; short names map branches, full refs map exactly. Repeatable. Not subject to --exclude-ref-prefix.
--tags
mirror tags (immutable by default — retarget needs a force flag)
--all-refs
every refs/* namespace (notes, pulls, custom). On sync/plan, also enables best-effort: per-ref target rejections become warnings, not failures. On replicate, strict.
--exclude-ref-prefix P
repeatable; drop refs under a prefix from discovery and prune scope. Classic: refs/pull/.
--prune
delete managed target refs that vanished from source (needs target delete-refs)

Safety — non-fast-forward

--force-with-lease
allow non-FF + tag retarget; reject if target moved during the run (per-run lease)
--force-blind
allow non-FF + retarget; overwrite regardless of current tip

Guardrails — bounding the work

--materialized-max-objects N
the key safety rail. Abort a non-relay materialized sync above N objects. Coarse, not a precise heap model — a backstop against the unbounded fallback.
--target-max-pack-bytes N
target receive-pack body limit; batches are planned & auto-subdivided to fit
--max-pack-bytes N
abort a bootstrap-relay push if the streamed source pack exceeds N bytes
--target-max-ref-updates N
max ref-update commands per push (0 = default 5000; env GITSYNC_MAX_REF_UPDATES_PER_PUSH)
--bootstrap-strategy
first-parent (default) or topo checkpoint chain ordering

Transport & visibility

--protocol auto|v1|v2
source-side protocol; push is always v1. Default auto.
--json
stable machine-readable output
--stats
per-service request/byte/want/have/command counters + throughput
--measure-memory
elapsed time + Go heap sampling
--progress / -v
live per-side throughput (TTY only) / verbose logging
--*-follow-info-refs-redirect
send follow-up RPCs to the final redirect host (per side)
--*-insecure-skip-tls-verify
self-signed local targets only — never the public internet
08 — Failure modes

What breaks, why, and the fix

Blocked: target not fast-forwardable

A branch diverged; the target tip isn't an ancestor of the source tip. Sync exits non-zero before any push.

Decide intent: --force-with-lease (safe overwrite) or --force-blind (unconditional). Or fix the divergence upstream.

Lease failure — target moved during sync

Under --force-with-lease, another writer moved the target between session start and push. receive-pack rejects "does not match expected old value." Fatal even under best-effort.

Rerun (re-captures the lease), or --force-blind if you truly mean to clobber.

Target rejected push — body too large

A relayed/batched pack exceeded the target's request-body limit.

Lower --target-max-pack-bytes (batched bootstrap auto-subdivides), or use a target with a higher limit.

Fell into materialized fallback unexpectedly

Something made relay ineligible: a force/prune flag, a delete, a tag retarget, or a no-thin target under sync. Memory + time spike.

Check relayReason in JSON. If it's no-thin and an overwrite is acceptable, replicate relays it instead.

replicate: requires relay-capable target

Replicate is relay-only by contract and refuses to materialize.

Rerun with sync (accepts the materialized fallback), or fix the target capability gap.

Materialized abort — too many objects

The fallback would exceed --materialized-max-objects. The guardrail did its job.

If legitimate, raise the limit after checking --measure-memory headroom; otherwise reshape the run (narrower scope, bootstrap the missing history first).

v2 negotiation failed

--protocol v2 was forced but the source returned v1 (old server, or a proxy stripping the Git-Protocol header).

Use --protocol auto to fall back gracefully.

Redirect chain mismatch (307 → 404)

The discovery GET redirected to host X, but an RPC POST hit a different redirect or expected sticky sessions.

Set --*-follow-info-refs-redirect so RPCs target the post-redirect host directly.

Auth 401 vs 403

401 = missing/wrong credentials (will retry via credential helper if configured). 403 = credentials accepted but lack permission for the action.

401 → fix the token/env/helper. 403 → grant the token write scope on the target (or read on a private source).

Source hidden-ref fetch refused (Gerrit refs/changes/*)

Best-effort only covers target-side receive-pack. If the source advertises a hidden ref but won't serve a want for it, the fetch errors out — no per-ref warn.

Scope it out with --exclude-ref-prefix (e.g. refs/changes/).

The mental model for triage Pack-level transport & unpack failures, and source upload-pack failures, are always fatal. Only target-side per-ref rejections can be downgraded to warnings — and only under best-effort (which --all-refs on sync turns on). Replicate never downgrades: its contract is "target == source."
